## 一、系统定位

一套面向Ozon俄罗斯电商平台的AI智能选品+修图+上架系统，核心解决"新卖家不知道怎么选品"和"选品到上架流程繁琐"两大痛点。

**一句话描述**：从选品发现到商品上架，AI做重活，人做决策。

## 二、业务架构

### 2.1 一期业务边界

```
选品发现 → 商品信息整理 → AI修图 → 上架
   ↑                            ↑         ↑
  AI推荐/深挖               人工审核   人工确认定价
```

一期做到上架为止，运营（自动调价/库存同步/客服）放二期。

> [V2.1补充] D-025选品历史复盘需要出单数据验证算法准确度，一期从现有ERP订单模块获取已上架SKU的出单数据（系统已有订单同步能力），不依赖二期运营模块。复盘一期聚焦「选品→上架→出单」完整链路。

### 2.2 核心业务流程

```

│                    ↓                                          │

### 2.4 人机协作边界

| 环节 | AI负责 | 人负责 |
|------|--------|--------|
| 选品方向 | 扫描数据、推荐机会 | 拍板选什么方向 |
| 选品推荐 | 输出Top N候选 | 挑选确认 |
| 信息生成 | 自动生成标题/描述/属性 | 可检查修正 |
| 修图 | AI自动处理 + 合规自动检查(8项) | **必须人工审核**（主观判断） |
| 定价 | AI给出建议价 | **必须人工确认**（SKU级） |
| 上架提交 | 自动调用API | 点击确认触发 |
| 状态追踪 | 自动轮询 | 被动通知 |
| 合规检查 | 自动检查EAC/禁售/知产 | 确认处理风险 |

**核心原则**：涉及对外展示（图片）和金额（定价）的事项，人把关。规则性检查（尺寸/水印/白底）机器自动做。

### 2.5 多店铺支持

一期即支持多店铺，API认证按店铺隔离 Client-Id / Api-Key。

## 三、核心业务对象

### 3.1 五大核心对象 [V2.1更新：新增SKU变体]

| 对象 | 说明 | 关键字段 |
|------|------|---------|
| **选品单** (Opportunity) | 记录一个选品机会 | 数据来源、品类、市场分析、利润估算、风险标记、状态、归属人 |
| **商品卡** (Product Card) | 准备上架的商品SPU级信息 | Ozon类目、标题、描述、SPU级属性、关联店铺 |
| **SKU变体** (Product Variant) [V2.1新增] | 商品的多规格变体 | 变体属性值、独立定价、库存、关联图片集 |
| **图片集** (Image Set) | 一个商品卡/变体的图片组 | 原图列表、修后图列表、主图、状态、合规检查结果、审核人 |
| **上架任务** (Listing Task) | 一次上架动作的记录 | 关联SKU、Ozon返回的task_id、审核状态、物流配置、失败原因 |

### 3.2 对象关系 [V2.1更新]

```
选品单 ──(确认选品)──→ 商品卡(SPU)
商品卡 ──(创建变体)──→ SKU变体
SKU变体 ──(关联)────→ 图片集
SKU变体 ──(触发上架)──→ 上架任务
店铺   ──(拥有)────→ 商品卡、上架任务
选品单 ──(转化)────→ 新选品单(精铺模式)  [V2.1新增：跟卖→精铺转化]
```

### 3.3 生命周期 [V2.1更新]

```
选品单:  发现 → 评估 → 确认/放弃 → [V2.1] 转化为精铺(创建新选品单)
商品卡:  创建 → 信息填充 → 变体创建 → 修图关联 → 待上架 → 已上架
SKU变体: 创建 → 定价确认 → 图片关联 → 待上架 → 已上架  [V2.1新增]
图片集:  创建 → AI修图 → 合规检查 → 待审核 → 已通过 → 已关联
上架任务: 创建 → 物流配置 → 提交中 → 审核中 → 成功/失败
```

## 四、选品模型算法 ⭐

> 详细设计文档：[选品模型算法设计V3](computer:///docs/ozon-selection-algorithm-v3.md)
> 算法研究报告：[选品算法研究](computer:///docs/Ozon选品算法研究_report.md)

### 4.1 六层递进架构（含第零层）

```
┌──────────────────────────────────────────────────────────────────┐
│  第零层：店铺阶段 + 卖家类型识别（决定硬约束规则集）               │
│    新店(≤3月)→跟卖模式/200~1500₽ | 老店(>3月)→精铺模式          │
│    中国卖家→EAC仅提示 | 俄罗斯本土卖家→EAC一票否决               │
│                                                                  │
│  第一层：硬约束过滤（按第零层规则集动态决定）                     │
│    EAC不合规(俄本土)→淘汰 / 新店200~1500₽→硬约束               │
│    体积>0.5L→降权 / 单价<500₽(精铺)→降权                        │
│                                                                  │
│  第二层：组合赋权（AHP主观 + 熵权法客观）                         │
│    品类×阶段自适应权重，跟卖/精铺用不同AHP判断矩阵                │
│                                                                  │
│  第三层：双通道评分                                               │
│    通道A：AHP-TOPSIS 多准则评分（结构化数据）                     │
│    通道B：[一期降级] Embedding余弦相似度 → [二期] LightGBM       │
│                                                                  │
│  第四层：Prophet时序预测                                          │
│    [一期冷启动] 类目级趋势替代 → [积累2周后] SKU级预测            │
│    内嵌俄罗斯假期日历，yearly_seasonality=15                      │
│                                                                  │
│  第五层：LLM推理层（信号解读 + 差异化建议 + 风险研判）            │
│    LLM只做解读不改分数                                            │
│                                                                  │
│  综合输出：复合评分 + 趋势方向 + AI解读 + 风险提示                │
└──────────────────────────────────────────────────────────────────┘
```

### 4.2 一期算法降级方案 [V2.1新增]

| 算法组件 | 一期方案 | 二期升级 | 触发条件 |
|---------|---------|---------|---------|
| **LightGBM语义评分** | Embedding余弦相似度（产品向量vs品类中心向量） | LightGBM回归模型 | 积累3个月标注数据 |
| **Prophet SKU级预测** | 类目级趋势（从知识库获取） | SKU级Prophet预测 | 积累2周SKU销量数据 |
| **pgvector向量检索** | jsonb存储+应用层余弦计算 | pgvector+HNSW索引 | 候选量>10万 |
| **综合评分权重** | TOPSIS 0.55 / 语义 0.20 / Prophet 0.10 / 机会指数 0.15 | TOPSIS 0.40 / 语义 0.20 / Prophet 0.25 / 机会指数 0.15 | Prophet SKU级可用时 |

### 4.3 学习闭环 [V2.1新增]

```
触发条件：
  ① 积累100条选品数据后触发首次回溯
  ② 之后每周自动回溯
  
回溯逻辑：
  对比AI评分等级 vs 实际表现(出单/利润)
  评分偏离>20% → 自动生成权重微调建议
  微调需人工确认后才生效（不自动改权重）
  
数据来源：
  selectionRetrospectives表记录每条选品的实际表现
  一期从ERP订单模块获取出单数据
```

## 六、数据源降级策略 [V2.1新增]

四源数据部分挂了系统怎么跑？三层降级方案：

### 6.1 降级等级

| 等级 | 触发条件 | 系统行为 | 用户可见 |
|------|---------|---------|---------|
| **L0 正常** | 4源均可用 | 完整四源交叉验证，折扣因子×1.0 | 无提示 |
| **L1 轻度降级** | 缺1源（非Ozon主源） | 三源交叉验证，缺失源对应端折扣 | "XX数据暂时不可用，评分置信度降低" |
| **L2 中度降级** | 缺2源（含Ozon主源） | 双源验证，评分打7折+标注"数据验证不足" | "Ozon API异常，仅用供给端数据评估" |
| **L3 严重降级** | 仅1源可用 | 单源评分，评分打5折+强制标注+人工确认 | "数据源严重不足，评分仅供参考，建议人工评估" |
| **L4 不可用** | 0源可用 | 暂停选品推荐，仅展示历史评分 | "数据源全部不可用，选品功能暂时不可用" |

### 6.2 降级自动恢复

```
dataSourceHealth表实时监控四源状态
  ↓
状态变化 → 自动切换降级等级
  ↓
恢复时 → 自动重算受影响的评分（标记为过期）
  ↓
通知用户"数据源已恢复，N条评分正在更新"
```

### 6.3 评分过期与重算

- 评分有效期7天（productScores.expiresAt）
- 降级期间产生的评分有效期缩短为3天
- 数据源恢复后，所有过期评分自动标记需重算
- 重算按优先级：A级> B级 > C级

## 九、跟卖→精铺转化流程 [V2.1新增，D-027]

```

  ① 店龄 > 3个月

     - 自动继承targetCategoryId/targetProductId
